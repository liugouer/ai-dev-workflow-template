# 知识库模板说明

## 概述

`.codebuddy/knowledge/` 是 CodeBuddy 理解项目业务的唯一信息源。每次启动研发闭环前，AI 都会读取所有知识库文件。

**关键原则：知识库内容必须是新项目的业务信息，不能照搬模板内容。**

## 爱穿搭项目知识库结构（参考）

```
.codebuddy/knowledge/
├── 01_项目概述.md      # 项目名称、目标、范围、原则
├── 02_架构设计.md      # 技术栈、分层架构、目录结构、接口
├── 03_核心模块.md      # 各模块职责、文件对应关系
├── 04_接口规范.md      # 接口路径、参数、响应格式、错误场景
└── 05_开发指南.md      # 环境搭建、依赖安装、启动命令、开发流程
```

## 模板：01_项目概述.md

```markdown
# 01 项目概述

## 项目名称
<你的项目名称>

## 项目目标
<一句话描述项目目标>

## 当前迭代
迭代：<迭代名称>
目标：<迭代目标>

## 核心原则
1. 每次只处理一个最小需求单元。
2. AI 不允许一次性扩展无关功能。
3. 所有实现必须先生成 Spec，再写代码。
4. 所有新增代码必须配套测试。
5. 测试覆盖率必须不低于 90%。
6. AI Review 评分必须不低于 95 分。
7. 有问题时 AI 必须自动修复并重新测试。
```

```markdown
## 安装依赖
<安装命令>

## 运行测试并生成覆盖率
<覆盖率命令>

## 开发流程
1. 从 TAPD 读取需求
2. 判断需求是否为最小需求单元
3. 生成 specs/{feature-name}-spec.md
4. 创建功能分支
5. 按 Spec 实现代码
6. 编写测试
7. 运行测试确保通过
8. AI Code Review
```

## 编写质量检查

| 检查项 | 要求 |
|--------|------|
| 每篇长度 | 精炼准确 |
| 代码示例 | 使用新项目的代码 |
| 目录结构 | 与当前实际一致 |
| 依赖版本 | 使用实际版本 |
| 密钥/Token | 不得出现 |
| 真实用户数据 | 不得出现 |
